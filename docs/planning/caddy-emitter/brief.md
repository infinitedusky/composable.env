# Brief: Caddy emitter — generate Caddyfile from contracts

Bookmarked 2026-05-11. Mirrors the existing nginx emitter; downstream
projects (numero) want to replace OrbStack's `*.orb.local` reverse proxy
with a Caddy container they control, to fix the stale-IP issue.

## Context from downstream

OrbStack's `*.<project>.orb.local` proxy caches upstream container IPs
at proxy-startup time. When a container restarts, the cached IP goes
stale and the browser sees 502 until `orb stop && orb start`. Caddy
running inside docker-compose, joined to the same network, re-resolves
Docker DNS per request — no stale cache.

Plan from numero side:
- One new `caddy-local` docker-compose service (regular `target.type:
  "docker-compose"` contract — no special framework support needed)
- Caddy mounts a generated Caddyfile via volume
- Each application contract declares its public hostname; ce builds
  the Caddyfile from the union of those declarations
- `*.<project>.local` resolves to 127.0.0.1 via `/etc/hosts`
- `caddy trust` makes the Caddy internal CA browser-trusted

## What composable.env adds

The existing nginx emitter already proves the pattern: per-contract
field → per-profile config file generated alongside `docker-compose.yml`
during `pnpm ce env:build`. The Caddy emitter is structurally identical.

### Contract field

Reuse the existing `target.subdomain` field — it already names the
public hostname. The generated address is `{subdomain}.{profile.domain}`
exactly like nginx. So `target.subdomain: "auth"` with profile domain
`numero.local` → `auth.numero.local`. Zero new contract fields.

The choice of emitter (nginx vs caddy vs both) is a profile decision,
not a per-contract one — declared in `ce.json`.

### ce.json

Add an optional `proxy` field to profile config:

```json
{
  "profiles": {
    "local": {
      "suffix": "-local",
      "domain": "numero.local",
      "proxy": "caddy"
    },
    "production": {
      "suffix": "",
      "domain": "chitin.casino",
      "proxy": "nginx"
    }
  }
}
```

Values: `"nginx"` (default — current behavior), `"caddy"`, or `"both"`.
Omitting it keeps today's behavior so existing projects are unaffected.

### Emitter

New file `src/targets/caddyfile.ts`:

```typescript
export function writeCaddyfile(
  projectRoot: string,
  contracts: Map<string, ServiceContract>,
  profileName: string,
  profileSuffix: string,
  profileDomain: string,
): CaddyfileResult
```

Same signature shape as `writeNginxConfigs()`. Output:

```caddyfile
# Caddyfile.local — Generated by composable.env

auth.numero.local {
  reverse_proxy auth-server-local:3673
}

poker.numero.local {
  reverse_proxy poker-local:3000
}

# ...one block per contract with subdomain set
```

Caddy's HTTPS is automatic (internal CA on `*.local` domains), no `tls`
directive needed unless we want to pin to specific cert files. The
emitter just emits hostname → upstream pairs.

### Builder integration

`buildServiceEnvironments` already calls `writeNginxConfigs` conditionally:

```typescript
if (profileConfigs) {
  // ...check for nginx profiles...
  writeNginxConfigs(...);
}
```

Add a parallel branch:

```typescript
if (proxy === 'caddy' || proxy === 'both') {
  writeCaddyfile(...);
}
if (proxy === 'nginx' || proxy === 'both' || !proxy) {
  writeNginxConfigs(...);  // default stays nginx for back-compat
}
```

### Auto-gitignore

`.gitignore` already handles `nginx.*.conf` via the marker block.
Add `Caddyfile.*` to the same block when proxy is caddy.

## Out of scope

- Caddy CA trust setup (`caddy trust`) — runs on the dev machine, not
  inside ce. Downstream project's `scripts/dev/trust-caddy.sh` owns it.
- `/etc/hosts` configuration — same, downstream project owns the setup
  script.
- TLS cert pinning — Caddy's auto-HTTPS via internal CA is the design;
  pinning would only matter if we wanted production parity (production
  uses nginx + Let's Encrypt, separate path).
- Caddyfile templates beyond simple `reverse_proxy` — middleware,
  rewrites, auth, etc. Out of scope until a use case demands it.
  The contract-driven approach generates the minimal viable Caddyfile.

## Tests

Mirror `tests/unit/nginx-emitter.test.ts` if it exists, or write fresh:

- Empty contracts → no Caddyfile written
- Contract with `target.subdomain` → emits one block with correct
  upstream (`{service}{suffix}:{port}`)
- Per-service overrides from `ce.json` `profiles[p].override[svc]`
  flow through correctly
- Multi-profile build emits one Caddyfile per profile
- `proxy: "both"` emits both nginx and caddy files
- Default (no proxy field) → nginx-only, no caddy emitted
