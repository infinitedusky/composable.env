# Brief: per-profile output filenames + arbitrary location paths

Bookmarked 2026-04-21. Not yet built.

## Problem

Today a contract's output file is always `${location}/.env.${profile}`. Two
real-world needs this doesn't cover:

1. **Custom filenames per profile.** Some tools expect specific filenames
   (`t.cyux.xy.env`, `local.env`, `service.env.prod`). The current pattern
   forces `.env.<profile>` and there's no way to override.
2. **Output anywhere on disk.** `location` is treated as relative to cwd,
   which technically allows absolute paths as a side effect (nothing rejects
   them) — but `~` doesn't expand, no docs say it's supported, and the
   filename suffix is still hardcoded.

## Sketch

Add an optional `outputs` map to the contract that maps profile name to
relative-or-absolute output path. Path is resolved relative to `location`
unless absolute. `~` expands to the user's home dir.

```json
{
  "name": "cyux",
  "location": "~/.config/cyux",
  "outputs": {
    "local": "t.cyux.xy.env",
    "production": "service.env.prod"
  }
}
```

Resolution rules:

1. If `outputs[profile]` is set, use it (relative to `location`, unless
   absolute or starts with `~`).
2. Else fall back to today's behavior: `${location}/.env.${profile}`.

`location` itself becomes:

- `apps/api` → relative to project (today's behavior, default)
- `/abs/path` → absolute
- `~/foo` → expand `~` to `$HOME`

Both `location` and `outputs[profile]` paths get the same expansion treatment.

## Backwards compatibility

Purely additive. Every existing contract keeps working with no changes:

- No `outputs` field → today's `${location}/.env.${profile}` behavior.
- `location: "apps/api"` → still relative to project root, same output path.
- `location: "/abs/path"` → already works as a side effect, just becomes
  documented.

Existing projects can adopt the new fields one contract at a time, or never.

## Why this matters

- Tools with non-standard filenames (Cypress fixtures, AWS Lambda layers,
  Vite envs without the `.env.` prefix, etc.) become first-class targets.
- Per-dev local config (e.g., personal toolchain configs in `~/.config/...`)
  can be generated by ce without polluting the project.
- Cleaner than the current "works as a side effect" absolute-path behavior.

## Open questions

- Validation: warn if an absolute output path escapes the project tree
  (without `~`)? Or just trust the user?
- `target` (docker-compose) interaction: targets always write inside the
  project. `outputs` only applies to `location`-style contracts.
- Profile-name-as-key vs default fallback: should `outputs.default` exist
  for the case where most profiles want a custom name but a few are fine
  with `.env.<profile>`?
- File deletion (CRU vs CRUD): tying into the broader "should ce ever
  delete files" question — outputs that go outside the project tree feel
  especially dangerous to ever delete automatically.
